# 企业知识库项目架构与功能说明

## 1. 项目定位

本项目是一个面向企业场景的多模态知识库后端系统，核心目标是将多来源文档统一转换、结构化存储并提供智能问答能力。系统已实现从基础 RAG 到 Agentic/Multi-Agent，再到 GraphRAG 与 RAGAS 评估的分阶段能力扩展。

当前技术底座：
- 后端：FastAPI + SQLAlchemy(Async) + Celery
- 存储：PostgreSQL + MinIO + Qdrant + Redis + Kuzu
- 模型：Claude（生成）、BGE-M3（Embedding）、BGE-Reranker（精排）

## 2. 总体架构

系统采用“API 层 + 领域服务层 + 基础设施层”的分层结构。

### 2.1 API 层（`app/api`）
- `documents.py`：文档上传、状态查询、内容读取、删除
- `search.py`：统一检索入口（向量/MCP/Agentic/Multi-Agent/Graph 增强）与 SSE 流式输出
- `mcp.py`：对外暴露 list/outline/search/read 四类工具接口
- `graph.py`：GraphRAG 图谱浏览、图查询、手动抽取触发
- `evaluation.py`：RAGAS 批量评估、CI 回归评估、历史对比

### 2.2 服务层（`app/services`）
- `conversion`：多模态格式转换与文档结构化入库
- `retrieval`：Embedding、向量索引、混合检索、路由、引用构建
- `agentic`：Plan-and-Execute 的 LangGraph 推理链
- `agents`：多 Agent 协作图（Orchestrator/Retrieval/Analysis/Generation）
- `graph`：实体关系抽取与 Kuzu 图存储
- `evaluation`：RAGAS 数据集构建、评估执行、调度
- `skills`：可插拔技能体系（分析、对比、健康检查、报告）
- `streaming`：SSE 事件流输出
- `mcp_tools`：MCP 四工具的具体实现

### 2.3 基础设施层（`app/core`）
- `config.py`：统一配置中心（环境变量驱动）
- `database.py`：异步数据库会话与事务边界
- `celery_app.py`：异步文档处理和定时评估任务
- `minio_client.py` / `qdrant_client.py` / `redis_client.py`：外部中间件封装

## 3. 核心数据模型

关键 ORM 模型（`app/models/database.py`）：
- `Project` / `ProjectMember`：项目与成员关系
- `Document`：文档主记录（状态、路径、校验、转换信息）
- `DocSection`：文档章节内容（检索和全文查询基座）
- `DocStructure`：目录树结构（大纲浏览）
- `AuditLog`：审计日志（问答日志、评估结果等）
- `TermGlossary`：术语词表（领域词扩展能力）

说明：`DocSection` 同时承载结构化内容、页码/Sheet/时间戳等溯源信息，满足多模态文档在检索与引用环节的定位需求。

## 4. 关键业务流程

## 4.1 文档入库流程（异步）
入口：`POST /api/v1/documents/upload`
1. 校验格式和大小，生成 checksum
2. 上传源文件到 MinIO
3. 写入 `documents`（`pending`）
4. 派发 Celery 任务
5. Worker 执行转换管道：
   - 源文件转 Markdown
   - 切分章节并写入 `doc_sections` / `doc_structure`
   - 写回 `documents` 状态为 `completed`
6. 向量索引写入 Qdrant（含 contextual enrichment）
7. 可选执行 GraphRAG 抽取（实体/关系入 Kuzu）

异常处理：
- 使用 `ProcessingArtifacts` 记录 MinIO/Qdrant 副作用，失败自动回滚
- Celery 任务包含重试、`acks_late`、`reject_on_worker_lost`，降低任务丢失风险

## 4.2 检索与问答流程
入口：`POST /api/v1/search/`
1. 查询路由（根据复杂度和文档规模选策略）
2. 执行策略：
   - `vector_rag`：Dense+Sparse 检索 + RRF + Reranker
   - `mcp_tool`：基于结构化章节的工具式检索
   - `agentic_rag`：Planner-Executor-Synthesizer 多步推理
   - `multi_agent`：多 Agent 并发/串行协作
   - `graph_enhanced`：图谱上下文增强向量检索
3. LLM 生成答案，注入引用
4. 写入 `audit_logs`
5. 按采样率触发在线 RAGAS 评估（异步）

## 4.3 SSE 流式问答
入口：`GET /api/v1/search/stream`
- 输出 `step/thinking/chunk/citation/done/error` 事件
- 支持前端实时显示“检索过程 + 增量答案”

## 5. 检索与智能能力实现细节

### 5.1 混合检索（`app/services/retrieval/searcher.py`）
- 向量编码：BGE-M3 生成 dense+sparse
- 检索执行：Qdrant dense 与 sparse 双通道
- 融合：RRF（加权）
- 精排：BGE-Reranker 对候选重排

### 5.2 路由策略（`app/services/retrieval/router.py`）
- 复杂问题：优先 `multi_agent`
- 关系型问题且 GraphRAG 启用：`graph_enhanced`
- 文档量阈值切换：大规模走 `vector_rag`，小规模走 `mcp_tool`

### 5.3 Agentic 与 Multi-Agent
- Agentic（Phase 2）：单图循环执行，强调步骤可解释和推理轨迹
- Multi-Agent（Phase 3）：星形拓扑，多角色协作，支持并行子任务

### 5.4 GraphRAG（`app/services/graph`）
- 抽取模块从章节内容中提取实体关系
- Kuzu 按 `project_id` 做数据隔离
- 支持邻居查询、实体类型统计、自然语言图查询

### 5.5 RAGAS 评估（`app/services/evaluation`）
- 每日批量评估（Celery Beat）
- 在线抽样评估（问答后异步触发）
- CI 回归评估（基于 golden dataset）

## 6. 对外接口概览

核心接口分组：
- 文档：上传、列表、详情、Markdown 获取、删除
- 检索：标准检索、流式检索、技能调用、技能列表
- MCP：文档列表、目录、章节搜索、章节读取
- 图谱：实体类型、邻居、图查询、抽取触发、项目图删除
- 评估：触发评估、CI 评估、历史查询、Run 对比
- 系统：`/` 根信息、`/health` 健康检查

## 7. 运行与部署

### 7.1 本地开发
1. `docker-compose up -d` 启动 PostgreSQL/Qdrant/MinIO/Redis（及可选 Celery 服务）
2. `pip install -r requirements.txt`
3. `alembic upgrade head`
4. `uvicorn app.main:app --reload --port 8000`
5. `celery -A app.core.celery_app worker --loglevel=info`（需要异步转换时）

### 7.2 容器编排
`docker-compose.yml` 已定义：
- `postgres`, `qdrant`, `minio`, `redis`
- `celery-worker`, `celery-beat`
- 持久化卷：`pgdata/qdrant_data/minio_data/redis_data/kuzu_data`

## 8. 配置项与开关

关键配置见 `app/core/config.py`：
- 数据源连接：PostgreSQL/Redis/MinIO/Qdrant
- 模型配置：`ANTHROPIC_MODEL`, `BGE_M3_MODEL_PATH`, `BGE_RERANKER_MODEL_PATH`
- 检索参数：`DENSE_WEIGHT/SPARSE_WEIGHT/COLBERT_WEIGHT/RERANKER_TOP_K`
- Agent 参数：`AGENTIC_MAX_STEPS`, `AGENT_GLOBAL_TOKEN_BUDGET`
- 功能开关：`CONTEXTUAL_RETRIEVAL_ENABLED`, `GRAPHRAG_ENABLED`, `RAGAS_ENABLED`

建议：按环境（dev/test/prod）维护独立 `.env`，避免模型和阈值在多环境耦合。

## 9. 当前实现特点与边界

已具备：
- 多模态文档处理和统一结构化存储
- 多策略 RAG 路由与流式问答
- Agentic/Multi-Agent 推理链路
- 图谱增强检索与评估闭环
- 异步处理与副作用回滚机制

当前边界：
- 认证鉴权与多租户安全边界未在当前代码中体现（接口多为开放调用）
- 观测告警能力主要依赖日志，尚未形成完整指标告警面板
- 文档与数据库迁移说明仍可进一步标准化（例如补充 API 示例与运维手册）
