# 后端待实现与差异说明

本文档与 [`前端接口对接清单.md`](./前端接口对接清单.md) 对应，归纳**当前 FastAPI 后端尚未提供或字段/路径不一致**的部分，便于排期与联调。实现状态以仓库 `app/api/`、`app/main.py` 为准。

---

## 1. 全局能力

| 项 | 状态 | 说明 |
|----|------|------|
| `Authorization: Bearer` 与 401 | **缺** | 全站无 JWT/会话校验，接口均为匿名可访问。 |
| 统一分页 `page` / `page_size` | **部分缺** | 列表类接口未统一分页约定。 |
| 结构化错误体 `code` / `field_errors` | **缺** | 多为 FastAPI 默认 `detail` 字符串。 |
| 按项目权限校验 | **缺** | 有 `project_id` 数据，无「当前用户能否访问该项目」逻辑。 |

---

## 2. 认证与用户（§1）— **整模块缺**

需新增：

- `POST /api/v1/auth/login`
- `POST /api/v1/auth/logout`
- `GET /api/v1/auth/me`
- `POST /api/v1/auth/refresh`（若采用双 Token）

依赖：用户表（或外部 IdP）、密码哈希、Token 签发/校验；并与下文 `upload_by`、成员角色等打通。

**与现有代码关系**：`POST /documents/upload` 当前强制表单字段 `upload_by`（UUID），对接后应改为从 Token 解析用户。

---

## 3. 项目与成员（§2）— **REST 缺（ORM 已有）**

数据库已有 `Project`、`ProjectMember`，但缺少 HTTP API：

| 路径 | 状态 |
|------|------|
| `GET/POST /api/v1/projects` | **缺** |
| `GET/PATCH/DELETE /api/v1/projects/{project_id}` | **缺** |
| `GET/POST /api/v1/projects/{project_id}/members` | **缺** |
| `PATCH/DELETE .../members/{user_id}` | **缺** |

**聚合/扩展字段缺**：列表项的 `phase`、`member_count`、`document_count`、`health`、`last_update_at`、`pending_summary`；详情的 `health` 对象、`timeline`、`last_report_excerpt`、`onboarding`（四布尔）。需在服务层统计或扩展存储。

**命名**：ORM 为 `stage`，清单为 `phase`，响应需对齐约定。

## 4. 文档与知识库（§3）— **部分缺 / 不一致**

### 4.1 缺失或需新增

| 清单要求 | 说明 |
|----------|------|
| `GET /api/v1/documents`（Query：`project_id`、`status`、`q`、分页） | 现有为 `GET /api/v1/documents/list/{project_id}`，路径与查询参数不一致。 |
| `GET /api/v1/documents/{id}/content` | 现有为 `GET /api/v1/documents/{id}/markdown`，可增别名或统一文档。 |
| `GET /api/v1/documents/{id}/status` | 现有可用 `GET /api/v1/documents/{id}` 轮询；清单若坚持独立轻量接口则需新增。 |
| 列表项 `progress`（0–100） | 当前 `DocumentInfo` 无该字段；需管道上报或推导规则。 |

### 4.2 请求/响应差异

| 项 | 清单 | 现状（摘要） |
|----|------|----------------|
| 上传 `project_id` | 可选 | 必填 |
| 上传 `upload_by` | 未列（应由登录态替代） | 表单必填 UUID |
| 上传响应 `document_id` | 示例字段名 | 返回 `doc_id` |
| 列表字段 `name` / `status` / `updated_at` | 清单 | `original_filename` / `conversion_status` / `upload_at` |

## 5. 检索与问答（§4）— **部分缺 / 不一致**

### 5.1 `POST /api/v1/search`（或 `POST /api/v1/search/`）

| 清单字段 | 现状 `SearchRequest` |
|----------|------------------------|
| `session_id`、`strategy`、`document_ids[]`、`complex` | **无** |
| `project_id` 可选 | **必填** |

响应清单需 `session_id`、可选 `steps`；现状为 `SearchResponse`（含 `results`、`latency_ms` 等），需扩展或增加前端适配 DTO。

### 5.2 引用结构 `Citation`

清单：`id`、`label`、`document_id`、`href`。  
现状：`CitationSource` 为扁平溯源字段（如 `doc_id`、`doc_name`、`section_path` 等），**需映射层或新响应模型**供 `CitationList` 使用。

### 5.3 `GET /api/v1/search/stream`

- 清单 Query：`session_id`、`strategy` —— 现状流式接口未对齐（且 `project_id` 在现有实现中为必填，清单为可选）。
- SSE 事件名（`step` / `thinking` / `chunk` / `citation` / `done` / `error`）与架构说明基本一致，**事件侧缺口较小**，主要补参数与会话关联。

## 6. 图谱 GraphRAG（§5）— **路径与响应形态不一致**

清单要求：

- `GET /api/v1/graph/projects/{project_id}/graph` → `nodes` / `edges`（React Flow）
- `POST /api/v1/graph/projects/{project_id}/query`（Body：`query`、`mode`）
- `POST /api/v1/graph/projects/{project_id}/extract`（触发抽取）

现状为另一套路由（如 `/graph/entities/{project_id}`、`GET /graph/query`、`POST /graph/extract/{doc_id}` 等），**无** `projects/.../graph` 子图聚合形态。需在图存储之上增加 **子图序列化** 与路径统一。
