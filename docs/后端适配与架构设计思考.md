# 后端适配思路与整体架构设计（讨论稿）

本文档在 [`后端待实现与差异说明.md`](./后端待实现与差异说明.md) 与 [`项目架构与功能说明.md`](../项目架构与功能说明.md) 基础上，归纳**如何分阶段适配前端**、**服务分层与边界**，以及 **Redis / Celery（消息队列）** 在现有工程中的角色与可扩展用法。

---

## 1. 适配总策略：先打通「身份与边界」，再补「业务模块」

### 1.1 原则

1. **不推倒重来**：在现有 FastAPI + 异步 SQLAlchemy + Celery + MinIO + Qdrant + Redis 栈上增量扩展。
2. **先横后纵**：先做 **认证、项目成员、权限中间件**，再做审批、通知、工作台等垂直业务；否则每条新接口都要返工补鉴权。
3. **契约优先**：与前端约定稳定的 **资源路径、分页、错误体、时间格式**；对已有接口采用 **兼容层**（别名路由、字段别名）而非一次性大破大立。
4. **计算与存储分离**：列表页需要的 `member_count`、`health`、`pending_summary` 等，优先 **查询时聚合** 或 **定时/事件增量刷新缓存**，避免在同步请求里做全库扫描。

### 1.2 建议阶段（与《前端接口对接清单》对接顺序一致）

| 阶段 | 目标 | 后端要点 |
|------|------|----------|
| A | 登录 + `me` + 项目 CRUD/成员 | User 表或 IdP；JWT；`Depends(get_current_user)`；项目 API 与 `ProjectMember.role` 校验 |
| B | 文档列表/上传/内容 | 路径与字段对齐；`upload_by` 从 Token 取；`progress` 策略见下文 |
| C | `POST /search` + SSE | 扩展 `SearchRequest`；Citation DTO；`session_id` 与会话表 |
| D | 审批 | 新表 + 状态机；**通过后**复用现有 Celery 文档处理或索引管线 |
| E | 工作台聚焦 + 通知 | 新表；**写操作同步、投递异步**（见 §4） |
| F | 图谱子图 API | 在 Kuzu 之上封装 `nodes/edges`；可与异步抽取任务配合 |
| G | 产出物、报告、协作 | 按演示优先级；报告生成宜 **异步任务** |

## 2. 整体架构（在现有三层之上扩展）

### 2.1 逻辑分层（保持现状，补齐缺口）

```
┌─────────────────────────────────────────────────────────────┐
│  API 层 (FastAPI routers)                                    │
│  auth / projects / documents / search / graph / approvals  │
│  workspace / notifications / artifacts / qa / reports        │
└───────────────────────────┬─────────────────────────────────┘
                            │ Depends: DB Session, Current User,
                            │         Project ACL
┌───────────────────────────▼─────────────────────────────────┐
│  领域服务层 (app/services/*) — 已有 + 新增                    │
│  conversion / retrieval / agentic / graph / skills / ...     │
│  + auth_service / project_service / approval_service / ...   │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────┐
│  基础设施层 (app/core/*)                                      │
│  PostgreSQL │ MinIO │ Qdrant │ Redis │ Celery broker/backend  │
└─────────────────────────────────────────────────────────────┘
```

**新增模块的典型位置**：

- **认证**：`app/core/security.py`（JWT 编解码、密码哈希）、`app/api/auth.py`、可选 `app/services/auth_service.py`。
- **项目/成员**：`app/api/projects.py` + `app/services/project_service.py`（聚合统计可放这里或 `repositories/`）。
- **横切**：`app/api/deps.py` 中统一 `get_current_user`、`require_project_role(...)`。

### 2.2 多租户与隔离

- 现有数据已带 `project_id`：**所有新项目接口**在业务层默认 `WHERE project_id = ? AND user_has_access(...)`。
- `org_id` 已在 `Project` 上：若毕设需要「组织」维度，登录用户绑定 `org_id`，项目列表再按组织过滤。

## 3. 消息队列与异步：现有能力 + 扩展建议

当前工程 **已使用 Celery**，Broker/Result 通常配置为 **Redis**（见 `CELERY_BROKER_URL`、`CELERY_RESULT_BACKEND`）。下面区分 **适合进队列** 与 **保持同步** 的工作负载。

### 3.1 已在队列中的（保持）

| 任务 | 说明 |
|------|------|
| `task_process_document` | 文档转换、章节入库、向量索引；**长耗时、不可阻塞 HTTP** |
| 定时 RAGAS 等 | `beat_schedule` 中已有每日评估类任务（按环境启用） |

**原则**：凡涉及 **大文件、多页解析、Embedding、批量写 Qdrant** 的，继续 **只通过 Celery** 执行；API 仅 **创建记录 + `delay()` + 返回 202/201**。

### 3.2 建议纳入队列的新增场景

| 场景 | 原因 | 实现要点 |
|------|------|----------|
| **审批通过 → 向量化** | 与上传入库同构，可能很重 | 审批接口只更新状态并入队 `process_document` 或 `reindex` 任务 |
| **报告生成** `POST .../reports/generate` | LLM 长文本 + 可能多文档聚合 | 异步任务写 `report` 表或对象存储；前端轮询或 WebSocket/SSE 推送「完成」 |
| **项目级 Graph 抽取** `POST .../extract` | 多文档批量抽取 | 任务内循环文档调用现有 `batch_extract` + Kuzu；API 返回 `job_id` |
| **批量通知**（如全员广播） | 避免单次请求超时 | 任务内写 `notifications` 表 |

### 3.3 建议保持同步（或仅短事务）

| 场景 | 原因 |
|------|------|
| **登录 / refresh / me** | 低延迟、强一致会话 |
| **项目与成员 CRUD** | 小事务，需立即返回 |
| **聚焦项、单条通知已读** | 简单 DB 更新 |
| **`POST /search`（非流式）** | 已在请求内完成；RAGAS 抽样已是 `asyncio.create_task` 式异步，可不进 Celery |
| **SSE `/search/stream`** | 长连接由 FastAPI `StreamingResponse` 处理，**不**用 Celery 跑流本身 |

### 3.4 Redis 的多角色（避免混淆）

| 用途 | 常见做法 | 在本项目中的含义 |
|------|----------|------------------|
| **Celery Broker** | 任务队列 | 与 worker 解耦，文档处理已依赖 |
| **Celery Result Backend**（若启用） | 任务结果查询 | 可选；简单场景可用 DB `job` 表代替 |
| **缓存** | 项目列表聚合、健康度摘要 | 可缓存 `GET /projects` 的计数类字段，TTL 30s～5min |
| **Pub/Sub**（可选） | 文档进度推送 | 架构说明曾提到「Redis pubsub 或轮询」；毕设可继续 **轮询 `GET /documents/{id}`**，降低复杂度 |
| **限流 / 会话黑名单** | JWT 登出、防刷 | refresh token 轮转或黑名单可用 Redis |

**建议**：毕设阶段 **优先轮询 + 明确 `progress` 规则**，Pub/Sub 作为增强项。

### 3.5 文档 `progress`（0–100）的工程化做法

- **实现成本从低到高**：  
  1. 仅按状态映射：`pending→0`、`processing→50`、`completed→100`、`failed→0`；  
  2. 在 Celery 任务关键步骤 `update_document(..., progress=…)`；  
  3. Worker 通过 Redis 写进度 + API 读取（更准，但更复杂）。

## 4. 跨模块一致性与「副作用」顺序

### 4.1 审批 → 文档可见性

- 明确：**未通过审批的文档**是否出现在 `GET /documents`、是否参与检索。
- 若「仅通过后入索引」：审批通过任务里再调 **与上传相同的索引管线**，并在 `Document` 上增加 `approval_status` 或在审批表中关联。

### 4.2 通知与审计

- **写审计**：关键操作（登录失败/成功、审批、删除文档、成员变更）同步写 `audit_logs`（已有表）。
- **发通知**：业务事务提交后 **`delay(notify_task)`**，避免「通知发出但 DB 回滚」。

### 4.3 会话 `session_id` 与检索

- 新增 `qa_sessions` / `qa_messages`（或等价表）；`POST /search` 创建或续写会话，返回 `session_id`。
- SSE 与同步检索共享同一套「会话上下文」逻辑，避免两套状态。

---

## 5. 图谱子图 API 的架构落点

- **存储**：继续使用 Kuzu（`app/services/graph/store`）。
- **新 HTTP 层**：在 `graph` 路由中增加清单约定的 `GET .../projects/{id}/graph`，服务内将 **图查询结果映射为 React Flow 的 `nodes`/`edges`**，与现有 `entities`/`neighbors` 并存或内部复用。
- **重计算**：大范围重抽仍走 Celery；**只读子图**走同步查询。

---

## 6. 部署与运维视角（毕设可简写）

- **进程**：至少 `uvicorn`（API）+ `celery worker`；生产可加 `celery beat`（已有 beat 配置）。
- **依赖顺序**：PostgreSQL、Redis、MinIO、Qdrant 就绪后再起 API 与 Worker。
- **健康检查**：现有 `/health` 已多组件探测；对齐前端时可在 `/api/v1/health` **做代理或别名**，避免监控改 URL。

---

## 7. 小结

| 维度 | 建议 |
|------|------|
| **适配节奏** | 鉴权与项目边界 → 文档与检索契约 → 审批与异步任务 → 工作台/通知 → 图谱与报告 |
| **消息队列** | 继续以 **Celery + Redis** 承载所有 **重计算与可重试** 任务；HTTP 保持轻量 |
| **Redis** | Broker + 可选缓存/限流；进度推送毕设可轮询优先 |
| **架构** | 在现有三层上 **加路由 + 加服务 + 加表**；避免在 API 函数内堆业务 |
